## 1. Общее описание

### 1.1 Назначение

AI-закупщик — цифровой сотрудник, автоматизирующий процессы закупочной деятельности ГШП:
- Подготовка и сопровождение закупок
- Анализ коммерческих предложений (КП)
- Помощь инициаторам в формировании закупочной документации

### 1.2 Заказчик

**Блоки-инициаторы:** БМТС, Блок Главного инженера / ОИСМ, Блок Главного инженера / ПБУ  
**Контактное лицо:** Макаров А.Б.

### 1.3 КПЭ

| Показатель | Целевое значение |
|---|---|
| Сокращение времени на подготовку и сопровождение закупок | 30–50% |
| Исключение дублирующих закупок | 100% |
| Оптимизация запасов МТР | 20–30% |

## 3. Проект 1: ИИ-помощник для закупщика

### 3.1 Описание

ИИ-ассистент, обученный на внутренних процедурах закупок ГШП. Помогает закупщику:
- Отвечать на типовые вопросы по закупочным процедурам
- Генерировать шаблоны документов
- Проверять соответствие заявок внутренним регламентам
- Находить аналогичные закупки в истории

### 3.2 Сложность и срок

| Параметр | Значение |
|---|---|
| Сложность | 3 |
| Срок реализации | 1–3 мес |
| Интеграция | 2.50 |

### 3.3 Функциональные требования

#### FR-1.1: Вопросно-ответный модуль

| Требование | Описание |
|---|---|
| FR-1.1.1 | ИИ должен отвечать на вопросы о процедурах закупок (ФЗ-223, ФЗ-44, внутренние ЛНА) |
| FR-1.1.2 | ИИ должен предоставлять ссылки на релевантные нормативные документы |
| FR-1.1.3 | ИИ должен уметь работать с контекстом предыдущих вопросов в рамках сессии |
| FR-1.1.4 | При невозможности ответа ИИ должен сообщать о неопределённости |

#### FR-1.2: Генерация документов

| Требование | Описание |
|---|---|
| FR-1.2.1 | ИИ должен генерировать шаблоны извещений о закупке |
| FR-1.2.2 | ИИ должен генерировать шаблоны технических заданий |
| FR-1.2.3 | ИИ должен генерировать шаблоны протоколов рассмотрения заявок |
| FR-1.2.4 | Сгенерированные документы должны соответствовать форматированию ГШП |

#### FR-1.3: Проверка заявок

| Требование | Описание |
|---|---|
| FR-1.3.1 | ИИ должен проверять заявки на соответствие требованиям извещения |
| FR-1.3.2 | ИИ должен выявлять типовые ошибки в заполнении заявок |
| FR-1.3.3 | ИИ должен формировать чек-лист проверки с указанием статуса по каждому пункту |

#### FR-1.4: Поиск по истории

| Требование | Описание |
|---|---|
| FR-1.4.1 | ИИ должен искать аналогичные закупки по ключевым словам |
| FR-1.4.2 | ИИ должен предоставлять информацию о ценах аналогичных закупок |
| FR-1.4.3 | ИИ должен уведомлять о потенциальных дублирующих закупках |

### 3.4 Нефункциональные требования

| Требование | Значение |
|---|---|
| Время ответа на вопрос | ≤ 10 сек |
| Доступность | 24/7 |
| Точность ответов ( baseline ) | ≥ 80% |
| Поддержка языка | Русский |

### 3.5 Интерфейс

- **Веб-интерфейс:** чат-бот в корпоративном портале
- **Входные данные:** текстовые запросы пользователя
- **Выходные данные:** текстовые ответы, сгенерированные документы

### 3.6 Моковые данные для тестирования

```
Вопрос: "Какой срок подачи заявок для закупки до 100 тыс. руб?"
Ответ: "Для закупок до 100 тыс. руб. (закупка у единственного поставщика)

Вопрос: "Создай извещение о закупке канцтоваров"

Ответ: "Проверка заявки № 12345:
```

## 4. Проект 2: Анализ коммерческих предложений

### 4.1 Описание

Автоматическое извлечение ключевых параметров из коммерческих предложений (КП):
- Цена, срок поставки, условия оплаты
- Гарантийные обязательства
- Соответствие техническим требованиям
- Ранжирование предложений по критериям

### 4.2 Сложность и срок

| Параметр | Значение |
|---|---|
| Сложность | 4 |
| Срок реализации | 1–3 мес |
| Интеграция | 2.00 |

### 4.3 Функциональные требования

#### FR-2.1: Извлечение данных из КП

| Требование | Описание |
|---|---|
| FR-2.1.1 | ИИ должен извлекать цену из КП в структурированном виде |
| FR-2.1.2 | ИИ должен извлекать срок поставки / выполнения работ |
| FR-2.1.3 | ИИ должен извлекать условия оплаты |
| FR-2.1.4 | ИИ должен извлекать гарантийные обязательства |
| FR-2.1.5 | ИИ должен извлекать информацию о доставке |
| FR-2.1.6 | ИИ должен поддерживать форматы PDF, DOCX, XLSX |

#### FR-2.2: Анализ соответствия

| Требование | Описание |
|---|---|
| FR-2.2.1 | ИИ должен сравнивать КП с техническими требованиями заказчика |
| FR-2.2.2 | ИИ должен выявлять несоответствия (отклонения от ТТ) |
| FR-2.2.3 | ИИ должен классифицировать несоответствия по степени критичности |
| FR-2.2.4 | ИИ должен формировать заключение о соответствии/несоответствии |

#### FR-2.3: Ранжирование предложений

| Требование | Описание |
|---|---|
| FR-2.3.1 | ИИ должен ранжировать КП по цене (от низкой к высокой) |
| FR-2.3.2 | ИИ должен учитывать весовые критерии при ранжировании |
| FR-2.3.3 | ИИ должен формировать рейтинг КП с обоснованием |
| FR-2.3.4 | ИИ должен рекомендовать оптимальное предложение с учётом всех факторов |

#### FR-2.4: Формирование отчётов

| Требование | Описание |
|---|---|
| FR-2.4.1 | ИИ должен формировать сводную таблицу по всем КП |
| FR-2.4.2 | ИИ должен формировать протокол рассмотрения КП |
| FR-2.4.3 | Отчёты должны быть в формате, пригодном для СЭД |

### 4.4 Нефункциональные требования

| Требование | Значение |
|---|---|
| Время обработки одного КП | ≤ 30 сек |
| Точность извлечения данных | ≥ 90% |
| Поддерживаемые языки КП | Русский, English |
| Обработка сканов | Требуется OCR |

### 4.5 Интерфейс

- **Загрузка КП:**drag-and-drop или выбор файлов
- **Ввод ТЗ:** ручной ввод или загрузка документа
- **Вывод:** таблица с результатами, протокол

### 4.6 Моковые данные для тестирования

**Входные данные (ТЗ):**
```
Закупка: Ноутбук Dell Latitude 5540
Количество: 10 шт.
Срок поставки: до 30.06.2026
Гарантия: не менее 3 лет
Бюджет: 1 500 000 руб.
```

**КП-1 (ООО "ТехноСнаб"):**
```

Срок поставки: 14 дней
Гарантия: 3 года
Условия оплаты: 50% аванс, 50% после поставки
```

**КП-2 (ИП Сидоров):**
```

Срок поставки: 7 дней
Гарантия: 1 год
Условия оплаты: 100% предоплата
```

**Результат анализа ИИ:**

|---|---|---|
| Цена | 1 350 000 ✅ | 1 200 000 ✅ |
| Срок поставки | 14 дней ✅ | 7 дней ✅ |
| Гарантия | 3 года ✅ | 1 год ❌ |
| Итого | **Соответствует** | **Не соответствует** |

**Рекомендация:** КП-1 (полное соответствие требованиям, оптимальное соотношение цена/качество)

## 5. Проект 3: ИИ-помощник для инициатора

### 5.1 Описание

ИИ-ассистент для сотрудников-инициаторов закупок, который:
- Проводит через «дерево решений» для корректного формирования заявки
- Помогает определить вид закупки и способ закупки
- Автоматически формирует пакет документов
- Проверяет комплектность заявки перед подачей

### 5.2 Сложность и срок

| Параметр | Значение |
|---|---|
| Сложность | 5 |
| Срок реализации | 3–6 мес |
| Интеграция | 1.50 |

### 5.3 Функциональные требования

#### FR-3.1: Дерево решений

| Требование | Описание |
|---|---|
| FR-3.1.1 | ИИ должен задавать уточняющие вопросы для определения типа закупки |
| FR-3.1.2 | ИИ должен определять применимый закон (ФЗ-223, ФЗ-44, внутризаводские) |
| FR-3.1.3 | ИИ должен рекомендовать способ закупки (открытый, закрытый, у ЕП и т.д.) |
| FR-3.1.4 | ИИ должен определять НМЦК или цену контракта |
| FR-3.1.5 | ИИ должен выявлять необходимость согласования с другими службами |

#### FR-3.2: Формирование документации

| Требование | Описание |
|---|---|
| FR-3.2.1 | ИИ должен генерировать заявку на закупку по форме ГШП |
| FR-3.2.2 | ИИ должен генерировать техническое задание |
| FR-3.2.3 | ИИ должен генерировать обоснование НМЦК |
| FR-3.2.4 | ИИ должен генерировать проект договора |
| FR-3.2.5 | ИИ должен заполнять все обязательные поля документации |

#### FR-3.3: Проверка комплектности

| Требование | Описание |
|---|---|
| FR-3.3.1 | ИИ должен проверять наличие всех обязательных документов |
| FR-3.3.2 | ИИ должен проверять корректность заполнения реквизитов |
| FR-3.3.3 | ИИ должен проверять соответствие ТЗ и заявки |
| FR-3.3.4 | ИИ должен формировать чек-лист готовности документов |

#### FR-3.4: Валидация и подсказки

| Требование | Описание |
|---|---|
| FR-3.4.1 | ИИ должен выявлять типовые ошибки при заполнении |
| FR-3.4.2 | ИИ должен предлагать исправления с объяснением |
| FR-3.4.3 | ИИ должен подсказывать, какие документы нужно получить дополнительно |
| FR-3.4.4 | ИИ должен предупреждать о нарушении сроков (если указана дата) |

### 5.4 Нефункциональные требования

| Требование | Значение |
|---|---|
| Время формирования пакета документов | ≤ 5 мин |
| Полнота проверки | ≥ 95% |
| Понятность подсказок | Для сотрудника без юридического образования |

### 5.5 Интерфейс

- **Диалоговый интерфейс:** пошаговый wizard
- **Ввод данных:** форма с полями, загрузка документов
- **Вывод:** готовый пакет документов, чек-лист

### 5.6 Моковые данные для тестирования

**Диалог с инициатором:**

```
ИИ: Для какой закупки вам нужна помощь?
Инициатор: Хочу закупить 20 ноутбуков для новых сотрудников

ИИ: Какова примерная стоимость?
Инициатор: Думаю, около 1.5 млн рублей

ИИ: Это плановая закупка или срочная?
Инициатор: Плановая, в рамках бюджета на этот год

ИИ: Есть ли утверждённая спецификация или нужно разработать техническое задание?
Инициатор: Нужно разработать ТЗ

    Рекомендую запрос предложений (быстрее).
```

```
ℹ️ Срок подачи заявки — не позднее 10.05.2026
```

## 6. Проект 4: Исполнение договора — анализ и контроль

### 6.1 Описание

ИИ-система для автоматического анализа и контроля исполнения договоров:
- Мониторинг сроков исполнения обязательств
- Контроль поставок и платежей
- Выявление отклонений от условий договора
- Формирование отчётов о статусе исполнения

### 6.2 Сложность и срок

| Параметр | Значение |
|---|---|
| Сложность | 3 |
| Срок реализации | 1–3 мес |
| Интеграция | 2.50 |

### 6.3 Функциональные требования

#### FR-4.1: Мониторинг договоров
