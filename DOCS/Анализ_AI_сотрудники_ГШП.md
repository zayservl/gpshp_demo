## 1. Платформа AI-сотрудников — единый продукт

AI-сотрудник — это **не отдельное приложение**, а роль на единой платформе-оркестраторе. Все направления работают на одном движке с одинаковым жизненным циклом задачи. Это ключевое решение: запуск одной роли автоматически даёт инфраструктуру для остальных.

### 1.1 Жизненный цикл задачи (пять шагов)

```
Делегирование  →  Уточнение  →  Исполнение  →  Результат  →  Коррекция
```

1. **Делегирование.** Пользователь ставит задачу естественным языком (текст / голос), опционально с прикрепленными документами.
2. **Уточнение.** Сотрудник задаёт уточняющие вопросы, если контекста недостаточно, и формирует план: что, какими инструментами, с какими источниками. План **согласуется** перед запуском.
3. **Исполнение.** Сотрудник работает, при необходимости в фоне. Пользователь видит прогресс в графе шагов, может вмешаться («стоп, возьми другие данные»).
4. **Результат.** Структурированный ответ со ссылками на источники, превью документов, таблицами, кнопками действий и **проактивными подсказками** — «что логично сделать дальше».
5. **Коррекция.** Пользователь вносит правки, даёт фидбек, либо нажимает предложенный шаг → задача продолжается без повторной постановки.

> **Главный инсайт.** Успех платформы = **минимизация когнитивной нагрузки**. Время от «постановки задачи» до «готового бизнес-результата» должно быть минимальным, а уровень доверия к результату достаточным, чтобы не перепроверять каждое действие.

### 1.2 Архитектурные принципы UX

| Принцип | Реализация |
|---|---|
| **Ролевой онбординг** | Профиль агента: должностные обязанности, инструменты, ограничения, быстрые действия (Quick Actions) |
| **Гибридный интерфейс (Chat + GUI)** | В диалог встроены таблицы, графики, превью документов, кнопки согласования — не только текст |
| **Асинхронность** | Задачу можно делегировать в фон («подготовь к утру») и получить уведомление; статус (Progress Tracking) доступен в любой момент |
| **Прозрачность мышления** | Видимая цепочка рассуждений: источники, шаги, выбранные инструменты, промежуточные выводы |
| **Со-авторство** | Пользователь может прервать выполнение, переиграть шаг, отредактировать черновик перед отправкой |
| **Интеграция с контекстом** | Доступ к внутренним системам (СЭД, 1С, ERP, базе НМД) в рамках прав пользователя |
| **Проактивность** | Сотрудник сам предлагает следующий шаг: «Я нашёл риск в §5.2 — подготовить протокол разногласий?» |

### 1.3 Handoff — сквозные цепочки между AI-сотрудниками

AI-сотрудник не изолирован: он может **передать задачу коллеге** с сохранением контекста. Это превращает портфель отдельных MVP в цепочки сквозных процессов.

**Типовые цепочки (реализованы в демо-прототипе):**

| № | Цепочка | Бизнес-процесс |
|---|---|---|
| 1 | Делопроизводитель → Юрист → Закупщик | Заявка прошла проверку комплектности → юридическая оценка условий → анализ КП и выбор подрядчика |
| 2 | Ассистент → Юрист | Обращение зарегистрировано → проект ответа → юридическая вычитка перед отправкой |
| 3 | Ассистент → Инженер ПБУ → Делопроизводитель | Суточная сводка → выявлен инцидент → маршрутизация регламентного письма |

Handoff включается одной кнопкой из проактивных подсказок, без потери контекста задачи.

## 12. Методология расчёта

- **Источник:** `Список_проектов_для_оценки_ГШП_ФИНАЛ.xlsx`, лист «Основной список», строки 4–271.
- **Фильтр MVP:** `Тип = "ИИ"` **И** `Оценка рисков/сложности = "Низкий"` **И** `Интегрированная оценка ≥ 1.4`.
- **Стоимость:** среднее значение диапазона из столбца «Оценка MVP Nedra» (напр. `"3-4 млн"` → `3.5 млн`). При единичном значении — само значение.
- **Волны по сложности:** `c = "Оценка сложности Nedra"`. Волна 1: c ≤ 4. Волна 2: c = 5–6. Волна 3: c ≥ 7.
- **Контроль расхождений.** Ранее в документах фигурировала цифра «39 MVP» — это соответствует фильтру «ИИ + низкий риск» **без** порога по интегральной оценке. В текущей версии используется строгий критерий заказчика (≥ 1.4), отсюда 17 проектов.
