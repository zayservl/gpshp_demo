## 1. Введение

### 1.1 Назначение документа

Данный документ определяет полные функциональные требования для MVP AI-делопроизводителя — цифрового сотрудника Газпром Шельф, предназначенного для автоматизации работы с документацией Общества.

### 1.2 Цель MVP

Демонстрация заказчику концепции AI-делопроизводителя через 2 ключевых сценария (P0):
1. Проверка комплектности заявок
2. Семантический поиск по документам (RAG)

### 1.3 КПЭ AI-делопроизводителя

| Показатель | Целевое значение |
|---|---|
| Ускорение поиска информации | в 3–5 раз |
| Сокращение времени на обработку корреспонденции | 30–50% |
| Снижение ошибок цитирования | 90%+ |

### 1.4 Блоки-инициаторы

- Блок Главного инженера / ОИСМ
- Блок Главного инженера / ПБУ
- БЭК
- Блок морской логистики
- БМТС
- Блок Главного инженера / УКК
- ОДОУ

### 1.5 Контактные лица

Воронков О.А., Паньков М.М., Майков Т.Г., Шпаченко И.П.

### Проект 1: Проверка комплектности и корректности заявок (P0)

**Сложность:** 4 | **Срок:** 1-3 мес | **Стоимость:** 4.0 млн

#### Описание

AI-агент проверяет заполненные заявки (закупки, поставки, ремонта, услуги) на наличие обязательных данных, выявляет ошибки и формирует рекомендации по исправлению.

#### FR-1.1: Приём заявки на проверку

| ID | Требование | Описание |
|---|---|---|
| FR-1.1.1 | Приём JSON | Система принимает заявку в виде структурированных данных (JSON) |
| FR-1.1.2 | Приём формы | Система поддерживает ввод через UI-форму |
| FR-1.1.3 | Приём API | Система поддерживает ввод через API-запрос |
| FR-1.1.4 | Идентификация типа | Система автоматически определяет тип заявки (postavka/zakupka/remont/usluga) |

#### FR-1.2: Извлечение и валидация полей

| ID | Требование | Описание |
|---|---|---|
| FR-1.2.1 | Извлечение полей | Система извлекает из заявки: наименование, количество, единица измерения, ТУ, согласования |
| FR-1.2.2 | Валидация заполненности | Для каждого поля выполняется проверка: заполнено / не заполнено |
| FR-1.2.3 | Валидация формата | Проверка корректности формата данных (числа, даты, email) |
| FR-1.2.4 | Проверка согласований | Система проверяет наличие и статус необходимых согласований |

#### FR-1.3: Формирование заключения

| ID | Требование | Описание |
|---|---|---|
| FR-1.3.1 | Отчёт о комплектности | Система формирует структурированный отчёт: заполнено / отсутствует / требует исправления |
| FR-1.3.2 | Рекомендации | Для каждой проблемы формируется рекомендация по исправлению |
| FR-1.3.3 | Оценка критичности | Система оценивает критичность каждого недостатка |

#### FR-1.4: AI-декомпозиция задачи

| ID | Требование | Описание |
|---|---|---|
| FR-1.4.1 | Task Decomposition | Система декомпозирует задачу на подзадачи |
| FR-1.4.2 | Визуализация графа | Отображается граф выполнения задачи |
| FR-1.4.3 | Логирование мышления | Отображается "мышление" агента: какие инструменты выбраны |

#### FR-1.5: Интерфейс проверки

| ID | Требование | Описание |
|---|---|---|
| FR-1.5.1 | Форма ввода | UI-форма для ручного ввода данных заявки |
| FR-1.5.2 | Загрузка JSON | Возможность загрузить файл JSON с заявкой |
| FR-1.5.3 | Результат в UI | Результаты отображаются в удобном для пользователя виде |

### Проект 2: Семантический поиск по ЛНА и ОРД (P0)

**Сложность:** 5 | **Срок:** 3-6 мес | **Стоимость:** 6.0 млн

#### Описание

RAG-система для семантического поиска по локальным нормативным актам и организационно-распорядительным документам. Поиск по смыслу, а не по ключевым словам.

#### FR-2.1: Индексация документов

| ID | Требование | Описание |
|---|---|---|
| FR-2.1.1 | Загрузка документов | Система загружает документы (PDF, DOCX, TXT) в векторную базу |
| FR-2.1.2 | Разбиение на чанки | Документы разбиваются на фрагменты (чанки) для индексации |
| FR-2.1.3 | Метаданные | Для каждого документа сохраняются: название, дата, тип, автор |
| FR-2.1.4 | Переиндексация | Возможность переиндексации при добавлении документов |

#### FR-2.2: Семантический поиск

| ID | Требование | Описание |
|---|---|---|
| FR-2.2.1 | Естественный язык | Пользователь вводит вопрос на естественном языке |
| FR-2.2.2 | Эмбеддинг запроса | Система преобразует запрос в векторную форму |
| FR-2.2.3 | Семантический поиск | Выполняется поиск по смыслу (не по ключевым словам) |
| FR-2.2.4 | Ранжирование | Результаты ранжируются по релевантности |

#### FR-2.3: Генерация ответа

| ID | Требование | Описание |
|---|---|---|
| FR-2.3.1 | Генерация ответа | LLM генерирует ответ на основе найденных фрагментов |
| FR-2.3.2 | Цитирование | Ответ включает точные цитаты из документов |
| FR-2.3.3 | Источники | Указываются источники: документ, страница, пункт |
| FR-2.3.4 | Confidence Score | Показывается уверенность в ответе (0-100%) |

#### FR-2.4: Интерфейс RAG

| ID | Требование | Описание |
|---|---|---|
| FR-2.4.1 | Чат-интерфейс | Пользователь вводит вопрос в чате |
| FR-2.4.2 | Визуализация источников | UI отображает найденные документы с подсветкой фрагментов |
| FR-2.4.3 | Переход к документу | Пользователь может кликнуть на источник и перейти к документу |
| FR-2.4.4 | История поиска | Сохраняется история поисковых запросов |

## 8. Ограничения и допущения

### 8.1 Ограничения MVP

1. **Моковые данные:** Все данные — тестовые, без интеграции с реальными системами ГШП
2. **Локальный LLM:** qwen3.5:2b имеет ограничения по качеству и размеру контекста
3. **Один пользователь:** Демо рассчитано на 1–3 одновременных пользователя
4. **Ограниченный RAG:** До 1000 страниц документов в индексе

### 8.2 Допущения

1. Заказчик имеет доступ к localhost для запуска
2. Демо проводится в офисе / переговорной с проектором
3. Заказчик понимает ограничения MVP (не Production)
4. Данные будут заменены на реальные после согласования

## 9. Глоссарий

| **RPA** | Robotic Process Automation — роботизация процессов |
