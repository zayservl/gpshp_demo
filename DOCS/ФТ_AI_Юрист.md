## 1. Общее описание

### Назначение системы

AI-юрист — цифровой сотрудник для автоматизации юридических процессов в ГШП:

- Проверка и корректура документов
- Перевод юридических текстов
- Анализ договорных условий
- Согласование договоров в СЭД
- Реагирование на нештатные ситуации

### Целевые пользователи

- Юристы ГШП
- Специалисты договорного отдела
- Руководители подразделений
- Аудиторы

### Ключевые показатели эффективности (КПЭ)

- Сокращение времени на обработку и согласование договорных документов: **40–60%**
- Снижение юридических рисков: **не менее 30%**
- Повышение качества документов: **не менее 85%**

### Технические ограничения (MVP)

- **LLM:** ollama qwen3.5:2b
- **Данные:** моковые (демонстрационные)
- **Интеграции:** без внешних систем (заглушки)

### Термины и сокращения

| Термін | Описание |
|---|---|
| ЛНА | Локальный нормативный акт |
| СЭД | Система электронного документооборота |
| КП | Коммерческое предложение |

## 3. Проект 1: Проверка текстов на опечатки и ошибки

### 3.1. Описание

**Назначение:** Автоматическая проверка юридических текстов на опечатки, орфографические ошибки, стилистические и типовые юридические ошибки.

**Сложность:** 3
**Срок:** 1–3 мес
**Стоимость:** 2.5 млн руб.

### 3.2. Функциональные требования (FR)

| ID | Требование | Описание | Приоритет |
|---|---|---|---|
| FR-1.1 | Загрузка текста | Система принимает текст для проверки (ввод через поле или загрузка файла .txt, .docx, .pdf) | P0 |
| FR-1.2 | Проверка орфографии | Система выявляет орфографические ошибки и опечатки | P0 |
| FR-1.3 | Проверка пунктуации | Система выявляет ошибки пунктуации | P0 |
| FR-1.4 | Проверка стиля | Система выявляет стилистические ошибки (канцеляризмы, тавтология) | P0 |
| FR-1.5 | Проверка типовых юридических ошибок | Система проверяет типовые ошибки в юридических текстах (неверное оформление дат, сумм, реквизитов) | P0 |
| FR-1.6 | Формирование отчёта | Система формирует отчёт с найденными ошибками, категориями и рекомендациями | P0 |
| FR-1.7 | Экспорт отчёта | Система позволяет экспортировать отчёт в файл (.docx, .pdf) | P1 |
| FR-1.8 | История проверок | Система сохраняет историю проверок пользователя | P2 |

### 3.3. Сценарии использования

#### Сценарий 1.1: Проверка договора

**Предусловия:** Пользователь имеет текст договора

**Основной поток:**

1. Пользователь вставляет текст в поле ввода или загружает файл
2. Пользователь нажимает кнопку «Проверить»
3. Система выполняет проверку (до 30 секунд)
4. Система отображает отчёт с ошибками
5. Пользователь просматривает отчёт

**Постусловия:** Отчёт отображён пользователю

#### Сценарий 1.2: Проверка письма

**Предусловия:** Пользователь имеет текст письма

**Основной поток:**

1. Пользователь вставляет текст в поле ввода
2. Пользователь выбирает тип документа «Деловое письмо»
3. Пользователь нажимает кнопку «Проверить»
4. Система выполняет проверку с учётом типа документа
5. Система отображает отчёт с рекомендациями

**Постусловия:** Отчёт отображён с учётом специфики письма

### 3.4. Моковые данные

#### Входные данные (пример 1)

```
ДОГОВОР № 125/2024

г. Москва «12» октября 2024 г.

Общество с ограниченной ответственностью «ТехноСтрой», именуемое в дальнейшем «Подрядчик», в лице директора Иванова А.С., действующего на основании Устава, с одной стороны, и Общество с ограниченной ответственностью «СтройМонтаж», именуемое в дальнейшем «Заказчик», в лице директора Петрова В.И., действующего на основании Устава, с другой стороны, заключили настоящий договор о нижеследующем:

1. ПРЕДМЕТ ДОГОВОРА
1.1. Подрядчик обязуется выполнить строительно-монтажные работы по объекту: «Реконструкция производственного цеха №3», расположенному по адресу: г. Москва, ул. Промышленная, д. 15, а Заказчик обязуется принять и оплатить результаты работ в порядке и на условиях, предусмотренных настоящим договором.
```

**Ожидаемые ошибки (мок):**

- «октября» → «октября» (правильно: «октября» — верно)
- «Устава» (кавычки) → лучше «Устава»
- Дата «12» октября — неполная форма (рекомендация: «12 октября 2024 г.»)
- Пункт 1.2 — отсутствует дата начала работ

### 3.5. Нефункциональные требования

| Точность обнаружения орфографических ошибок | ≥ 90% |

## 4. Проект 2: Перевод юридических текстов

### 4.1. Описание

**Назначение:** Автоматический перевод юридических текстов с сохранением смысла, единой терминологии и юридической точности.

**Сложность:** 4
**Срок:** 1–3 мес
**Стоимость:** 4.5 млн руб.

| ID | Требование | Описание | Приоритет |
|---|---|---|---|
| FR-2.1 | Загрузка текста для перевода | Система принимает текст для перевода (русский → английский или наоборот) | P0 |
| FR-2.2 | Определение языка | Система автоматически определяет исходный язык | P0 |
| FR-2.3 | Перевод текста | Система выполняет перевод с сохранением смысла | P0 |
| FR-2.4 | Сохранение терминологии | Система использует единую юридическую терминологию | P0 |

| FR-2.7 | Глоссарий терминов | Система отображает глоссарий использованных терминов | P1 |
| FR-2.8 | Сравнение версий | Система позволяет сравнить оригинал и перевод | P2 |

### 4.3. Сценарии использования

#### Сценарий 2.1: Перевод договора

**Предусловия:** Пользователь имеет текст договора на русском языке

**Основной поток:**

1. Пользователь вставляет текст договора
2. Пользователь выбирает направление перевода (RU → EN)
3. Пользователь нажимает «Перевести»
4. Система выполняет перевод (до 60 секунд)
5. Система отображает перевод с подсветкой терминов

**Постусловия:** Перевод отображён пользователю

#### Сценарий 2.2: Перевод письма партнёру

**Предусловия:** Пользователь имеет деловое письмо

**Основной поток:**

1. Пользователь загружает файл письма
2. Система определяет язык автоматически
3. Пользователь подтверждает направление
4. Система переводит с учётом делового стиля
5. Пользователь может редактировать перевод

**Постусловия:** Перевод готов к отправке

## 5. Проект 3: Анализ подходов служб к условиям договоров

### 5.1. Описание

**Назначение:** Классификация и анализ замечаний разных служб к условиям договоров, выявление типовых подходов и противоречий.

**Сложность:** 5
**Срок:** 3–6 мес
**Стоимость:** 6.0 млн руб.

### 5.2. Функциональные требования (FR)

| ID | Требование | Описание | Приоритет |
|---|---|---|---|
| FR-3.1 | Загрузка замечаний | Система принимает текст замечаний или таблицу замечаний | P0 |
