# 🏭 Aurora.MIND · Платформа AI-сотрудников ГШП

Демонстрационный прототип платформы **AI-сотрудников** для АО «Газпром Шельфпроект» (ГШП).
Каждый сотрудник — это автономный агент с профилем, набором инструментов, готовыми сценариями и
диалогом «как с коллегой»: делегирование → уточнение → **план-граф** → исполнение → результат с проактивными следующими шагами.

Ключевые особенности демо:

- **Plan Graph** — интерактивный редактируемый граф плана, который пользователь видит и может править **до** запуска: удалить шаг, добавить паузу, поменять параметры, сохранить как шаблон.
- **Handoff между AI-сотрудниками** — результат одного сотрудника одним нажатием передаётся другому (закупщик → юрист, юрист → закупщик, делопроизводитель → юрист) с сохранением контекста и, по желанию, с конкретным сценарием у принимающего.
- **Пауза / Продолжить** — можно остановить исполнение после любого шага; в момент паузы сразу появляется промежуточный документ-снимок в панели «Документы», чтобы пользователь видел осязаемый артефакт, а не только чат.

## ✨ Возможности

- **4 готовых AI-сотрудника** — Делопроизводитель (Анна), Юрист (Александр), Закупщик (Николай), Ассистент.
- **Сценарии «под ключ»** — от проверки комплектности заявки и юр. проверки КП перед подписанием до протокола закупочной комиссии по ремонту ППБУ.
- **Пятишаговый диалог** — делегирование, уточнение, план-предложение, исполнение с трассировкой, результат + «что дальше».
- **Интерактивный Plan Graph** (ReactFlow) — в центре рабочей области до запуска; после одобрения сменяется на execution-граф с live-статусами.
  - Удалить шаг (`removed`) → будет пропущен.
  - Поставить паузу после шага (`pause_after`) → исполнение остановится и запросит подтверждение.
  - Отредактировать параметры узла (номер договора, тендер, период и т. п.).
  - Включить опциональный handoff-узел в конце графа.
  - Сохранить текущий план как **шаблон** и подставить его в будущем.
- **Handoff задач** — проактивные кнопки `· handoff` переключают активного сотрудника и могут сразу указать целевой `scenario_id` у принимающего (минуя keyword-классификатор).
- **Промежуточные документы** — при паузе и по завершении сценариев автоматически появляются артефакты в панели «Документы» (правовое заключение, протокол, shortlist подрядчиков, реестр контрактов, промежуточный отчёт и т. д.).
- **Свободный ввод** — если запрос не попал в готовый сценарий, отвечает LLM в роли сотрудника, с предложением быстрых пресетов.
- **Визуализация оркестрации** — граф шагов плана, логи рассуждений, список созданных документов.
- **Просмотр документов** — все моки (заявки, договоры, тендеры, КП, письма, обращения, ЛНА, ежедневные сводки) открываются во встроенном viewer.
- **Real-time UI** — обновления идут через WebSocket (`chat_message`, `workflow_update`, `node_status`, `document_created`, …), с fallback-опросом сессии.

- `Делопроизводитель → Юрист`: проверка заявки №127 → передача на правовой анализ договора на ремонт.
- `Юрист → Закупщик`: анализ рисков Д-2026-001 → подбор альтернативных подрядчиков (`find_alternative_contractors`).

- `Ассистент → Юрист`: проект ответа на обращение → проверка формулировок (`proofread_draft`).

## 🚀 Быстрый старт

### 1. Требования

- **Python 3.11+** (проверено на 3.9/3.11).
- **Node.js 20+** и `npm`.
- **Ollama** с моделью `qwen3.5:2b` (дефолт). Модель легко меняется флагом `--model`.

### 2. Установка Ollama и модели

```bash
# macOS / Linux
curl -fsSL https://ollama.com/install.sh | sh

# запуск Ollama (в отдельном окне, если не поднят автоматически)
ollama serve

# загрузка модели по умолчанию
ollama pull qwen3.5:2b
```

### 3. Запуск одной командой

```bash
./start.sh
```

Скрипт:

1. проверит Ollama и модель;
2. создаст `.venv` и поставит python-зависимости;
3. поставит `node_modules` для `frontend/`;
4. освободит порты **8000** и **5173** от зависших процессов прошлого запуска;
5. поднимет backend (FastAPI) и frontend (Vite);
6. выведет URL сервисов.

Флаги:

```bash
./start.sh --reload              # backend с авто-перезагрузкой (dev)
./start.sh --skip-deps           # быстрый рестарт без переустановки зависимостей
./start.sh --model qwen3:8b      # использовать другую модель Ollama
```

После запуска откройте:

| Что | URL |
| --- | --- |
| UI платформы | http://localhost:5173 |
| REST API | http://localhost:8000 |
| Swagger / OpenAPI | http://localhost:8000/docs |
| Health | http://localhost:8000/health |

Остановка — `Ctrl+C` в терминале со `start.sh` (скрипт корректно убьёт оба процесса).

### 4. Ручной запуск (без start.sh)

```bash
# Зависимости
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
(cd frontend && npm install)

# Backend (терминал 1)
PYTHONPATH=. python -m uvicorn backend.main:app --host 0.0.0.0 --port 8000 --reload

# Frontend (терминал 2)
cd frontend && npm run dev
```

## 🧪 Демо-сценарий «как показать за 7 минут»

1. **Открываем `http://localhost:5173`** → витрина из 4 AI-сотрудников.
2. **Выбираем Анну (Делопроизводитель)** → быстрые сценарии слева.
3. Нажимаем `📋 Проверить заявку №125`:
   - Анна предлагает **план-граф** в центре экрана → видно 5 шагов и опциональный handoff-узел.
   - В чате — краткая сводка плана с кнопкой «Открыт в центре ↗».
   - (опционально) Можно удалить шаг, поставить паузу, подменить номер заявки прямо в графе, затем нажать **«Согласовать и выполнить»**.
   - После approve центр переключается на **execution-граф** с анимацией live-статусов.
   - В результате — карточка заявки, найденные замечания, созданный отчёт о комплектности.
4. В блоке «Следующие шаги» нажимаем **«Запустить маршрут согласования»**.
   - Плавно переходим к follow-up сценарию: маршрут, уведомления согласующим, статус 4 из 4.
5. Возвращаемся к результату проверки заявки и нажимаем жёлтую кнопку с меткой `· handoff` — **«Передать юристу для правового анализа договора на ремонт»**.
   - Платформа автоматически **переключает активного сотрудника на Александра (Юрист)**, показывает приветствие «Принял задачу от коллеги — Анна…» и сразу предлагает план анализа Д-2026-002.
6. В результатах юриста жмём **«Подготовить протокол разногласий»** → получаем сформированный документ.
7. Переходим к **Закупщику Николаю** → `🛠️ Сравнить КП по технадзору БК-3` (`zkp_2026_028`) → уточнение → план:
   - В плановом графе включаем handoff-узел `Передать юристу на юр. проверку` (по умолчанию он `removed`).
   - Ставим **паузу после шага 3** (оценка подрядчиков) — сразу после паузы в «Документах» появляется **промежуточный отчёт** со списком выполненных и оставшихся шагов.
   - Жмём **«Продолжить»** → план добегает → рейтинг трёх КП с рекомендацией победителя.
8. В проактиве кликаем `· handoff` → **юрист сразу запускает сценарий `check_kp_legal_risks`** (не требует keyword-классификации) → юридическое заключение по КП с рисками по пунктам 5.2 / 7 / 11 и протоколом разногласий.
9. Переходим к **Ассистенту** → `📅 Ежедневная сводка за 21.04` → `📊 Сравнить со сводкой за 20.04` → таблица с динамикой KPI по платформам.
10. Любой документ (в том числе сгенерированный ИИ и промежуточные снимки с паузы) открывается кликом в правой панели «Документы».

### Шаблоны плана

Если демо-сценарий повторяется регулярно (например, «юр. проверка КП» с конкретной комбинацией параметров/пауз), отредактированный план-граф можно **сохранить как шаблон** прямо из тулбара над графом и в следующий раз подставить его одной кнопкой — без ручных правок.

## 🔌 API

Полная спецификация — `http://localhost:8000/docs`. Ключевые эндпоинты:

| Метод | Endpoint | Описание |
| --- | --- | --- |
| GET | `/api/employees` | Список сотрудников для витрины |
| GET | `/api/employees/{id}` | Профиль сотрудника (инструменты + сценарии) |
| POST | `/api/chat/{id}/init` | Инициализировать чат-сессию с сотрудником |
| POST | `/api/chat/{id}/reset` | Сбросить сессию |
| GET | `/api/chat/{id}` | Текущее состояние сессии (сообщения, `pending_plan`, `documents_created`) |
| POST | `/api/chat/{id}/message` | Отправить сообщение (`text`, `scenario_id?`, `handoff_from?`) |
| POST | `/api/chat/{id}/approve` | Подтвердить предложенный план (запускает материализацию: `removed` пропускаются, `pause_after` → паузы, активный handoff → proactive) |
| POST | `/api/chat/{id}/reject` | Отклонить план |
| POST | `/api/chat/{id}/plan/update` | Сохранить правки планового графа (`graph_nodes`, `graph_edges`) |
| POST | `/api/chat/{id}/plan/resume` | Снять активную паузу и продолжить исполнение |
| POST | `/api/chat/{id}/plan/template` | Сохранить текущий план как шаблон (`name`) |
| GET | `/api/chat/{id}/plan/templates` | Список шаблонов этого сотрудника |
| POST | `/api/chat/{id}/plan/template/{template_id}/apply` | Подставить шаблон как `pending_plan` |
| GET | `/api/documents?employee_id=…` | Реестр документов |
| GET | `/api/documents/{doc_id}` | Документ + разобранное содержимое (`content_full`) |
| GET | `/health` | Health check |
| WS | `/ws` | WebSocket: `chat_message` (в т. ч. `plan_proposal`, `plan_paused`, `plan_resumed`, `result`), `workflow_update`, `node_status`, `log`, `agent_reasoning`, `document_created`, `workflow_completed` |

Handoff на стороне API — это просто `POST /api/chat/{target_id}/message` с `scenario_id`
из `proactive[].scenario_id` и `handoff_from` = id исходного сотрудника. Фронт делает это автоматически,
когда нажимают кнопку с `target_employee_id`. Если в плановом графе активен handoff-узел с
`handoff_scenario_id`, у принимающего сотрудника сразу запускается именно этот сценарий —
без keyword-классификации.

Все моки лежат в `data/*.json` и загружаются «лениво» при первом обращении через `DataStore`.
Чтобы добавить сценарий или сотрудника — достаточно отредактировать `data/employees.json`
и (при необходимости) добавить инструмент в `backend/tools/registry.py` +
сборщик результата в `backend/chat/result_builder.py`. Шаблоны плана хранятся **в памяти процесса**
(`backend/chat/plan_templates.py`) и живут только до рестарта backend — для демо этого достаточно.

## 🐛 Частые проблемы

**Ollama не подключается:**

```bash
curl http://localhost:11434/api/tags      # проверка
ollama serve                              # перезапуск
```

**Порт 8000 / 5173 занят:** `./start.sh` освобождает оба порта перед запуском. Если нужно вручную:

```bash
lsof -ti:8000 | xargs kill -9
lsof -ti:5173 | xargs kill -9
```

**Фронт не подхватывает изменения:** Vite HMR обычно справляется, но после правок `App.jsx` / handoff-логики
сделайте хард-рефреш вкладки (`Cmd+Shift+R`). Backend без `--reload` нужно перезапускать явно.

**WebSocket не соединяется:** проверьте, что backend поднят на `:8000` и CORS в `backend/main.py` разрешает
`http://localhost:5173`.
